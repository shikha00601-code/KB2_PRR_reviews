PR #31: Implement order cancellation with saga pattern
Author: Priya Nair
Date: 2025-09-15
Status: Merged
Branch: feature/order-cancellation-saga

Description:
Implement coordinated order cancellation across services using 
choreography-based saga pattern via Kafka events.

Review Comments:

Rahul Sharma (Sep 16):
"Why choreography over orchestration? With choreography, the flow 
is implicit in the event chain. If we need to add a step later 
(like sending a cancellation email), every developer needs to 
understand the full event flow to know where to add it."

Priya Nair (Sep 16):
"For our current flow (cancel order -> release inventory -> refund), 
choreography is simpler. We already have Kafka events flowing. Adding 
an orchestrator service adds another moving part to maintain. If our 
cancellation flow gets more complex (5+ steps), I agree we should 
switch to orchestration. For now, choreography keeps it lean."

Karan Joshi (Sep 17):
"What happens if the refund step fails but the inventory is already 
released? We'd have inconsistent state."

Priya Nair (Sep 17):
"Good edge case. Added a REFUND_FAILED event. When this happens, 
the order stays in CANCELLATION_PENDING state and we alert the ops 
team. The inventory stays released (correct behavior — we don't want 
to re-reserve it). The refund gets retried manually or through the 
DLT replay mechanism Rahul is building."

Final status: Approved by Rahul, Karan. Merged Sep 18.
